## 2. 现状与目标差距

| 目录 | 目标框架 | 实际现状 | 差距 | 迁移动作 |
|------|----------|----------|------|----------|
| [weiji-web](weiji-web/) | 功能参考原型 | 纯静态 HTML+JS，**功能最全**（30+ 业务、完整数据模型、7 页面） | 无需改造 | 标注为原型，加 README |
| [weiji-server](weiji-server/) | cool-admin-midway | **自研** Koa + Midway 风格装饰器（非真 cool-admin），11 控制器 + 12 张 MySQL 表 + 36 测试 | 框架需替换 | 用 cool-admin-midway 脚手架重建，迁移 entity/service/controller |
| [weiji-admin-web](weiji-admin-web/) | cool-admin-vue | Vue3 + Element Plus **标准栈**（非 cool-admin-vue），9 页面 + 25 测试 | 需换脚手架 | 用 cool-admin-vue 脚手架重建，迁移页面 + 用 cl-crud 重写管理页 |
| [weiji-app](weiji-app/) | cool-uni | uni-app（Vue3.4，微信小程序为主），8 页面 | 需对齐 cool-uni 模板 | 用 cool-uni 脚手架重建，迁移页面 + 接入 Service 自动化 |
| [weiji-ai](weiji-ai/) | （独立 AI 层） | FastAPI，6 厂商集成，21 测试，降级完善 | 无需改造 | 保持原样，仅被新 server 调用 |

**核心结论：** 现有三个工程都不是真正的 cool-admin 全家桶。cool-admin 是完整脚手架（自带 RBAC、菜单、字典、日志、文件管理、CRUD 代码生成、插件体系），目录约定与启动方式和现有自研代码差异大，"渐进改造"不可行——本质是用官方脚手架重建 + 业务迁移。

### 3.2 端口与技术栈

| 服务 | 端口 | 框架 | 技术栈 | 职责 |
|------|------|------|--------|------|
| weiji-server | **8001** | cool-admin-midway | Midway.js + TypeORM + MySQL + Redis + `@cool-midway/core` | 业务后端 + 后台管理 API（含 base 权限体系） |
| weiji-admin-web | **9000** | cool-admin-vue | Vue3 + Vite + Element Plus + cl-crud/cl-form | PC 后台管理 + 业务页面 |
| weiji-app | **5173(H5)** | cool-uni | uni-app + Vue3 组合式 + Pinia | 移动端（微信小程序 / H5 / App） |
| weiji-ai | **8002** | FastAPI | Python + httpx + AsyncOpenAI | 独立 AI 服务（不变） |

> 端口 8001/9000 对齐 cool-admin 官方默认，降低认知负担。原 admin-web 的 5173 让渡给 weiji-app 的 H5 调试。

### 3.3 API 分层（关键设计）

cool-admin 按路由前缀天然分层，味记沿用：

| 前缀 | 用途 | 鉴权 | 消费方 |
|------|------|------|--------|
| `/admin/*` | **B 端后台管理**：base 模块（用户/角色/菜单/字典/日志/文件）+ 业务管理页（record/family 等的 CRUD 管理） | cool-admin token + RBAC 权限 | cool-admin-vue 后台 |
| `/app/*` | **C 端业务 API**：味记 App 用户用的业务接口（记录/家庭/成就/打卡/AI 等） | App 端 JWT | weiji-app、admin-web 业务页 |
| `/open/*` | **开放接口**：无需登录的公开数据（如健康检查、公开菜谱） | 无 | 任意 |

> ⚠️ **破坏性变更：** 现有接口统一在 `/api/*` 下（如 `/api/record/list`）。迁移后 C 端业务接口路径变为 `/app/record/...`，**所有前端请求路径需同步修改**（见 [§8.5](#85-api-路径契约变更破坏性)）。

### 5.2 模块划分（业务域映射）

现有 11 个控制器按业务域归并为 **7 个 cool-admin 模块**：

| cool-admin 模块 | 含义 | 对应现有控制器/能力 |
|-----------------|------|---------------------|
| `record` | 美食记录 | RecordController + 相关表 records |
| `family` | 家庭组域 | FamilyController（家庭/成员/邀请/菜谱/菜单/购物/动态/报告） |
| `achievement` | 成就与等级 | AchievementController + ChallengeController（可拆分） |
| `checkin` | 打卡 | CheckinController |
| `gamification` | 趣味玩法 | GamificationController（图鉴/人格/时光机/盲猜） |
| `ai` | AI 代理 | AiController → 转发 weiji-ai:8002 |
| `analytics` | 数据埋点 | AnalyticsController |

> `auth` 与 `user` 不建独立模块：B 端用户复用 `base/sys/user`；C 端 App 用户在 `record` 或新建 `account` 模块下建 `app_user` 实体（见 [§5.7](#57-c-端用户-vs-b-端用户分离)）。

### 5.3 Entity 写法范式

依据 `@cool-midway/core` v7.x 真实范式（`BaseEntity` 自带 `id`/`createTime`/`updateTime`，内置软删除 `deleteTime`）：

```typescript
// src/modules/record/entity/record.ts
import { BaseEntity } from '@cool-midway/core';
import { Column, Entity, Index } from 'typeorm';

@Entity('weiji_record')               // 建议统一 weiji_ 前缀
export class RecordEntity extends BaseEntity {
  @Index()
  @Column({ comment: '用户ID', type: 'bigint' })
  userId: number;

  @Column({ comment: '菜品名称', length: 100 })
  dishName: string;

  @Column({ comment: '烹饪方式', length: 20, nullable: true })
  cookingMethod: string;

  @Column({ comment: '评分1-5', type: 'tinyint', default: 3 })
  rating: number;

  @Column({ comment: '营养分析', type: 'json', nullable: true })
  nutrition: any;

  @Column({ comment: '餐次', length: 20, default: '' })
  mealType: string;

  @Index()
  @Column({ comment: '记录日期', type: 'date' })
  recordDate: string;

  @Column({ comment: '来源', length: 20, default: 'manual' })
  source: string;
}
```

> - `BaseEntity` 已含 `id`(自增 bigint)、`createTime`、`updateTime`、`deleteTime`(软删除)，无需再手写 `createdAt`/`updatedAt`/`isDeleted`
> - JSON 字段（ingredients/steps/nutrition/condition/rules）用 `type: 'json'`
> - `synchronize: true` 自动建表，**生产环境务必关闭**（[§11](#11-风险与待确认决策)）

### 5.5 配置项映射

| 现有配置（weiji-server） | 迁移目标 | 位置 |
|--------------------------|----------|------|

### 5.6 cool-admin 能力复用（节省的开发量）

| cool-admin 内置能力 | 味记应用场景 |
|---------------------|--------------|
| **base 模块 RBAC** | B 端后台管理（用户/角色/菜单/权限）开箱即用；家庭组权限（owner/admin/member）在 `family` 模块自建轻量角色映射，不强行套 RBAC |
| **文件管理** | 美食照片、菜谱封面上传（替代现有自研上传） |
| **数据回收站** | 记录/菜谱软删除后可恢复（替代现有 `isDeleted`） |
| **操作日志** | 后台操作审计 |
| **定时任务**（`@cool-midway/task` + BullMQ） | 连续打卡检测、家庭饮食报告生成、挑战赛结算 |
| **cl-crud 自动生成** | record/family/recipe 等管理页一行 CRUD，省去手写增删改查 |
| **字典管理** | 餐次、烹饪方式、难度、购物分类等枚举值 |

### 5.7 C 端用户 vs B 端用户分离

**关键设计：分离两套用户体系。**

| 用户类型 | 实体 | 存储 | 鉴权 | 说明 |
|----------|------|------|------|------|
| **B 端管理员** | `base_sys_user`（cool-admin 内置） | MySQL | cool-admin token + RBAC | 后台运营人员，量小，权限细 |
| **C 端 App 用户** | `weiji_app_user`（新建 entity） | MySQL | App 端独立 JWT | 味记 App 普通用户，量大，字段含昵称/头像/家庭组关联 |

理由：C 端用户字段（昵称、头像、家庭组角色）与 B 端管理用户差异大；cool-admin RBAC 对 C 端过重；分离后 C 端鉴权独立、可水平扩展。现有 `users` 表数据迁移到 `weiji_app_user`（id 从 uuid 映射为 bigint）。

## 8. 三端集成与契约

### 8.1 统一响应格式

三端沿用现有契约，**cool-admin 原生即此格式，天然兼容**：

```ts
// 成功：{ code: 0, data: <payload>, message: "" }
// 失败：{ code: <非零>, data: null, message: "人类可读" }
```

- HTTP：鉴权失败 401，参数错误 400，资源不存在 404，其余成功 200
- AI 降级时 HTTP 200，由 `code` 区分业务失败

### 8.2 鉴权（JWT）

- **B 端**：cool-admin base token（`/admin/base/open/login` 签发，含 RBAC）
- **C 端**：App 端独立 token（`/app/account/login` 签发，绑定 `weiji_app_user.id`）
- 前端各端拦截器自动注入 Authorization、401 清理重定向（cool-admin-vue / cool-uni 均内置）

### 8.3 AI 调用链路

```
weiji-app / admin-web ──/app/ai/*──▶ weiji-server(ai模块)
                                          │ HttpPost + 30s 超时 + 健康检查
                                          ▼
                                     weiji-ai:8002 (/ai/*)
                                          │ 降级：key缺失/调用失败 → mock；违规 → code:1
```

- `ai` 模块 service 封装转发逻辑（迁移现有 `AiProxyService`），处理 multipart 与 JSON
- 健康检查：60s 轮询 weiji-ai `/health`，不可用时 `/app/ai/*` 返回 `code:503`
- AI 厂商 Key 仍由 weiji-ai 通过环境变量管理，**不硬编码**

### 8.4 端口与代理

| 工程 | 开发代理 |
|------|----------|
| weiji-admin-web | `vite` 代理 `/admin`、`/app` → `http://localhost:8001` |
| weiji-app（H5） | `vite` 代理 `/app` → `http://localhost:8001`；小程序直连 `http://localhost:8001/app`（需关闭合法域名校验） |
| weiji-server | 反向代理 `/api/ai/*` → `http://localhost:8002`（或 ai 模块直接调用） |

### 8.5 API 路径契约变更（破坏性）

| 现有路径 | 新路径 | 说明 |
|----------|--------|------|
| `POST /api/auth/login` | `POST /app/account/login` | C 端登录 |
| `POST /api/auth/register` | `POST /app/account/register` | — |
| `GET /api/record/list` | `GET /app/record/list` 或 `/app/record/page` | 分页可走 cl-crud 的 page |
| `POST /api/record` | `POST /app/record/save` | — |
| `GET /api/family/recipes` | `GET /app/family/recipe/list` | family 域统一前缀 |
| `POST /api/ai/recognize` | `POST /app/ai/recognize` | ai 模块 |
| `GET /health` | `GET /open/health` 或保留 `/health` | 公开 |
| （B 端管理） | `POST /admin/record/page` 等 | cl-crud 自动生成 |

> 所有前端 API 调用点需批量改路径。建议 Phase 0 先定一份《API 路径映射表》作为契约，三端按表同步。
